## D001: OnCaveChanged の差分更新を後回し

**日付**: 2026-03-21
**フェーズ**: Phase 2-C

**判断**: `ChiFlowEngine.OnCaveChanged` は龍脈を全再計算する実装とした。差分更新（変更された部分のみ再計算）は行わない。

**理由**:

- 現時点では部屋数・龍脈数が少なく（数十オーダー）、全再計算でもパフォーマンスに問題がない
- 差分更新は「どの龍脈が影響を受けたか」の追跡が必要で、複雑さが大幅に増す
- Phase 3 以降で仙獣の配置変更による気の再計算頻度が上がった時点で、ボトルネックになるか計測してから最適化する

**影響範囲**: `fengshui/chi_flow.go` の `OnCaveChanged` メソッド

**将来の対応**: Phase 7（統合シミュレーション）でプロファイリングを行い、必要に応じて差分更新に切り替える

## D002: コアゲーム体験の定義 — 「常に不完全な状態での判断」

**日付**: 2026-03-21
**フェーズ**: Phase 3（全フェーズに影響する設計原則）

**判断**: カオスシードの面白さの核心は「完璧なダンジョンを完成させること」ではなく、「常に不完全な状態で判断を迫られ、80%の妥協で噛み合わせるプロセス」にある。この体験を保証するため、以下の3つをシステムレベルで設計原則とする。

### 原則1: 不完全性の強制（Phase 1/6 に影響）

理想のレイアウトが物理的に実現不可能であること。

- 地形にランダムな制約を導入する（掘削不可の硬質岩盤、地下水脈、既存の空洞など）
- 洞窟生成時にRNGで障害物を配置し、毎回異なる「パズル」にする
- 龍脈の源泉位置もランダムで、理想の風水配置が地形ごとに変わる
- **影響**: Phase 1 の Grid/Cave に地形バリエーション追加が必要（Phase 4以降で対応）

### 原則2: 時間圧力（Phase 4/6 に影響）

侵入波がプレイヤーの準備完了を待たないこと。

- 侵入波のタイミングはシナリオ側で固定スケジュール（プレイヤー進捗非連動）
- 序盤は弱い侵入者が早い段階で来る → 最低限の防衛を優先するか構築を優先するかの判断
- 中盤以降は構築フェーズと防衛フェーズが重なる頻度を上げる
- 「全部屋完成してから防衛開始」というプレイを不可能にする
- **影響**: Phase 4 の侵入波生成と Phase 6 のシナリオ設計で、タイミング調整が最重要パラメータとなる

### 原則3: トレードオフの連続（Phase 5 に影響）

リソース（気）が常に不足し、全方面への投資が不可能であること。

- 気の収入（龍脈）は有限で、以下の用途が競合する:
  - 新しい部屋の掘削コスト
  - 既存部屋の強化
  - 仙獣の育成（気を消費して成長）
  - 仙獣の新規召喚コスト
  - 罠の設置・維持
- 1つに投資すると他が手薄になる構造を維持する
- 「全部の部屋がレベルMAX、全仙獣がレベルMAX」はゲームクリア前には到達不能なバランス
- **影響**: Phase 5 の経済バランスは「余裕がない」状態をデフォルトとして設計する

### アンチパターン（避けるべき設計）

- 十分な時間があれば完璧なダンジョンが完成する → ゲームの死
- 最適解が1つに収束し、毎回同じ配置が最強 → リプレイ性の喪失
- 侵入者が弱すぎて構築の邪魔にならない → 判断の必要性が消滅
- リソースが潤沢で全部に投資できる → トレードオフの消滅

### 検証方法

Phase 7（統合シミュレーション）で以下を検証する:

1. AIプレイヤーによる数千ゲームの自動実行で、「全部屋MAX」到達率がゲームクリア前に0%に近いこと
2. 同じシナリオでも地形RNGの違いで最適配置が変わること
3. 侵入波の50%以上が「構築中」（部屋掘削/仙獣配置の最中）に到達すること

### 備考

この判断はオリジナルのカオスシードのプレイ体験に基づく。具体的なパラメータ（地形障害物の密度、侵入波の間隔、リソースの逼迫度）はPhase 7のシミュレーションで調整する。重要なのはパラメータの値ではなく、3つの原則がシステム構造として組み込まれていることである。

# DECISIONS.md D002 追記分 — 人間から見たゲームの面白さの言語化

## 面白さの構造分解（プレイヤー体験の観点）

以下はオリジナルのカオスシードで実際に感じた面白さを、体験の種類ごとに言語化したもの。
システム設計の判断に迷ったときは、「この設計はどの面白さを生むのか、あるいは殺すのか」で検証する。

### 1. 計画する面白さ（構築フェーズ）

- 地形を見て「ここにこの部屋を作れば最高のレイアウトになるな」と構想する楽しさ
- 龍脈の位置と部屋の属性を考えて、相生の連鎖を頭の中で組み立てる知的快感
- 「理想の完成形」が見えている状態そのものが心地よい
- **設計への示唆**: 地形が読みやすく、計画を立てたくなる情報がプレイヤーに見えていること。風水の仕組みが直感的に理解できること

### 2. 妥協する面白さ（制約との対話）

- 理想通りにはいかない。地形の制約で「あ、でもこっちの部屋はこうなっちゃったんだ」と気づく瞬間
- でも悲観ではなく「じゃあ別の部屋をこう作ってこう繋げると8割ぐらいの満足度になるぞ」と代替案を見つける喜び
- 100点は取れないけど80点の解を見つける「パズルを解いている」感覚
- **設計への示唆**: 制約は「理不尽な壁」ではなく「別解を探す動機」であるべき。常に80点の妥協案が存在する地形生成が望ましい。0点（詰み）も100点（理想完遂）も起きてはいけない

### 3. 中断される面白さ（構築 × 防衛の衝突）

- チクチクとレイアウトを構築している最中に敵が攻めてくる
- 構築中だから防衛も中途半端。理想のレイアウトの途中なので理想の防御はできない
- この「両方とも不完全な状態で同時にやりくりする」忙しさが楽しい
- **設計への示唆**: 構築フェーズと防衛フェーズが明確に分離してはいけない。両方が重なるタイミングがゲームのピーク体験。シナリオ設計の最重要要素

### 4. 噛み合う面白さ（創発的成功）

- 不完全な防衛なのに、なんとか噛み合って撃退できた瞬間のカタルシス
- 仙獣の配置がたまたまいい位置にいた、部屋の属性がたまたま有利だった、そういう「計画半分・偶然半分」の成功
- 自分の判断が報われた感覚と、運が味方した感覚の混合
- **設計への示唆**: 戦闘結果に適度なランダム性が必要。100%計算で勝てる戦闘はつまらないし、100%運の戦闘は理不尽。「正しい配置をした上で、ダイスが味方する」バランス

### 5. 完璧が来ないことの面白さ（永続的不完全性）

- 完璧な部屋ができるとそこでゲームは終わる。判断が要らなくなるから
- 面白いのは「完璧を目指すプロセス」であって「完璧な状態」ではない
- ゲーム終了時に「もうちょっとやりたかった」「次はもっとうまくやれる」と思える状態が理想
- **設計への示唆**: ゲームクリア条件は「完璧なダンジョン完成」ではなく「N波の侵入を耐え凌ぐ」等の生存系。クリア時点でもダンジョンは未完成で、「次こそは」のリプレイ動機が残る

## 面白さの連鎖モデル

上記の5つは独立ではなく、以下の順序で連鎖する:

```
計画する → 制約に出会う → 妥協する → 構築を始める
    ↑                                       ↓
    └──── 「次こそは」のリプレイ動機 ←── 防衛で中断される
                                            ↓
                                    不完全な状態で噛み合わせる
                                            ↓
                                    撃退成功のカタルシス
                                            ↓
                                    でも完璧ではない → 次の計画へ
```

このループが1ゲーム内で何度も回り、かつゲーム間でも回ることがリプレイ性を生む。

## パラメータ調整時の判断基準

Phase 7 のシミュレーションやプレイテストで調整を行う際、以下を基準とする:

| 面白さの種類       | 壊れるサイン                         | 調整の方向                                |
| ------------------ | ------------------------------------ | ----------------------------------------- |
| 計画する面白さ     | プレイヤーが計画を立てる前に敵が来る | 最初の侵入波までの猶予を伸ばす            |
| 妥協する面白さ     | 常に理想配置が可能 / 常に詰む        | 地形制約の密度を調整                      |
| 中断される面白さ   | 構築と防衛が完全に交互               | 侵入波の間隔をランダム化・構築中に重ねる  |
| 噛み合う面白さ     | 戦闘が常に圧勝 / 常に惨敗            | 侵入者の強さ曲線とリソース供給を調整      |
| 完璧が来ない面白さ | ゲームクリア前に全MAXに到達          | リソースの総量を絞る / クリア条件を早める |

## D003: 仙獣の気消費は GrowthEngine 側で RoomChi.Current を直接減算する方式

**日付**: 2026-03-21
**フェーズ**: Phase 3-D

**判断**: 仙獣の成長時の気消費は、ChiFlowEngine に消費APIを追加するのではなく、GrowthEngine が `RoomChi.Current` を直接減算する方式とした。

**理由**:

- ChiFlowEngine は気の「流れ」（供給・伝播・減衰）に責務を持ち、消費は別レイヤーの関心事
- GrowthEngine が消費量を計算して即座に適用する方がロジックの流れが追いやすい
- RoomChi.Current は公開フィールドであり、外部からの操作を許容する設計
- 将来的に仙獣以外（罠の維持、部屋の強化等）も気を消費するため、消費元ごとに自分で減算する方式が拡張しやすい

**影響範囲**: `senju/growth.go` の Tick メソッド

**将来の対応**: Phase 5（経済システム）で気の収支を一元管理する仕組みを導入する場合、消費を記録するイベントログ的な仕組みの追加を検討する

## D004: 仙獣の戦闘ステータスは配置時ではなくクエリ時に計算

**日付**: 2026-03-21
**フェーズ**: Phase 3-C

**判断**: `Beast.CalcCombatStats(roomChi)` はキャッシュせず、呼び出し時に毎回計算する方式とした。

**理由**:

- RoomChi は毎ティック変動するため、キャッシュの無効化タイミングが複雑になる
- 戦闘ステータスの計算は乗算数回で非常に軽量
- 戦闘解決時（Phase 4）にのみ呼ばれるため、頻度も低い

**影響範囲**: `senju/combat_stats.go`

**将来の対応**: Phase 4 の戦闘システムで多数の仙獣が同時戦闘する場合でも、計算量はO(N)で問題にならない見込み

## D005: 仙獣AIの行動衝突は先着順で解決

**日付**: 2026-03-21
**フェーズ**: Phase 3.5-C

**判断**: BehaviorEngine の Tick メソッドで、複数の仙獣が同じ部屋に同時移動しようとした場合、先着順（beasts スライスの順序）で解決する方式とした。

**理由**:

- 優先度ベースの衝突解決（SPD順、レベル順等）は直感的だが、仙獣の順序が毎ティック変わるため決定論的再現が難しくなる
- 先着順はシンプルで決定論的。beasts スライスの順序は呼び出し側が制御できる
- 衝突が頻発する状況は部屋数に対して仙獣数が多い場合に限られ、実際のゲームプレイでは稀

**影響範囲**: `senju/behavior_engine.go` の Tick メソッド

**将来の対応**: Phase 4 で侵入者と仙獣の移動衝突が加わる際に、優先度ベースへの変更を検討する

## D006: 仙獣の行動AIは侵入者位置をmap[int][]intで受け取る

**日付**: 2026-03-21
**フェーズ**: Phase 3.5-A

**判断**: BehaviorEngine.Tick の侵入者位置引数は `map[int][]int`（部屋ID → 侵入者IDリスト）とした。Phase 4 の Invader 型に依存しない。

**理由**:

- senju パッケージが invasion パッケージに依存すると循環依存の原因になる（依存方向: senju ← invasion）
- 侵入者の「どの部屋にいるか」だけが行動判定に必要であり、侵入者の詳細情報（HP、属性等）は不要
- Phase 4 で侵入者型を定義した後、invasion 側が senju の BehaviorEngine を呼び出す際にこのマップを構築する

**影響範囲**: `senju/behavior_engine.go`, `senju/behavior.go` の BehaviorContext

**将来の対応**: Phase 4 で戦闘解決時に侵入者の詳細情報が必要になったら、BehaviorContext に追加のフィールドを設ける

## D007: D002時間圧力のPhase 4段階の実装

**日付**: 2026-03-21
**フェーズ**: Phase 4

**判断**: D002で定義した「時間圧力」原則を Phase 4 で以下の形で具体化した:

1. **WaveSchedule によるスケジュール固定**: 侵入波の発生タイミングは `TriggerTick` で事前定義。プレイヤーの構築進捗とは無関係に到来する
2. **テスト用スケジュールの早期侵入**: tick 50 での最初の侵入波は、十分な構築ができていないタイミングを意図的に設定
3. **難易度のエスカレーション**: 波ごとに Difficulty が上昇し、侵入者数・レベルが増加。後半の波は前半より明確に強い

**シナリオレベルの調整（Phase 6 への申し送り）**:
- WaveSchedule はシナリオシステムで上書きされる前提。Phase 4 のスケジュールはあくまでテスト用
- D002の「構築中に防衛が割り込む」体験は、Phase 6 でシナリオごとの波間隔チューニングで実現する
- 重要なのは Phase 4 が「波のタイミングを外部から注入できる」設計になっていること

**影響範囲**: `invasion/wave_schedule.go`, `invasion/wave_generator.go`

## D008: 戦闘マッチングは素早さ順ペアリング・1ティック1ラウンド

**日付**: 2026-03-21
**フェーズ**: Phase 4-C

**判断**: 同じ部屋に複数の仙獣と侵入者がいる場合の戦闘マッチングを以下のルールで実装した:

1. **素早さ（SPD）順でペアリング**: 全戦闘参加者を SPD 降順でソートし、最速の仙獣と最速の侵入者をペアリング、次のペア…と順にマッチ
2. **余剰は戦闘なし**: 仙獣数 ≠ 侵入者数の場合、余った側はそのラウンドでは戦闘しない（フリー）
3. **1ティック1ラウンド**: 各ペアの戦闘は先攻→後攻の1往復のみ。複数ラウンドの連続戦闘はない

**理由**:
- 多対多の戦闘マッチングは組み合わせ爆発の原因になりうるが、SPDソート→順序ペアリングは O(N log N) で決定論的
- 1ティック1ラウンドはティックベース設計と整合的。毎ティック状況が変化する（移動、撤退、新たな侵入者の到着）ため、長時間の戦闘ロックは不適切
- 余剰フリーのルールにより、数的優位がそのまま「攻撃を受けない」メリットになり、防衛配置の戦略性が生まれる

**代替案の棄却**:
- ランダムペアリング: 決定論的ではない（RNG を使えば再現可能だが、SPD という既存のステータスで自然に決まる方が良い）
- 全員同時攻撃: 余剰の概念がなくなり、数的優位のメリットが薄れる
- 複数ラウンド: ティック間で移動・撤退の判断ができなくなり、D002の「判断の連続」体験を阻害する

**影響範囲**: `invasion/combat.go` の `ResolveRoomCombat`

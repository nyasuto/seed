## D001: OnCaveChanged の差分更新を後回し

**日付**: 2026-03-21
**フェーズ**: Phase 2-C

**判断**: `ChiFlowEngine.OnCaveChanged` は龍脈を全再計算する実装とした。差分更新（変更された部分のみ再計算）は行わない。

**理由**:

- 現時点では部屋数・龍脈数が少なく（数十オーダー）、全再計算でもパフォーマンスに問題がない
- 差分更新は「どの龍脈が影響を受けたか」の追跡が必要で、複雑さが大幅に増す
- Phase 3 以降で仙獣の配置変更による気の再計算頻度が上がった時点で、ボトルネックになるか計測してから最適化する

**影響範囲**: `fengshui/chi_flow.go` の `OnCaveChanged` メソッド

**将来の対応**: Phase 7（統合シミュレーション）でプロファイリングを行い、必要に応じて差分更新に切り替える

## D002: コアゲーム体験の定義 — 「常に不完全な状態での判断」

**日付**: 2026-03-21
**フェーズ**: Phase 3（全フェーズに影響する設計原則）

**判断**: カオスシードの面白さの核心は「完璧なダンジョンを完成させること」ではなく、「常に不完全な状態で判断を迫られ、80%の妥協で噛み合わせるプロセス」にある。この体験を保証するため、以下の3つをシステムレベルで設計原則とする。

### 原則1: 不完全性の強制（Phase 1/6 に影響）

理想のレイアウトが物理的に実現不可能であること。

- 地形にランダムな制約を導入する（掘削不可の硬質岩盤、地下水脈、既存の空洞など）
- 洞窟生成時にRNGで障害物を配置し、毎回異なる「パズル」にする
- 龍脈の源泉位置もランダムで、理想の風水配置が地形ごとに変わる
- **影響**: Phase 1 の Grid/Cave に地形バリエーション追加が必要（Phase 4以降で対応）

### 原則2: 時間圧力（Phase 4/6 に影響）

侵入波がプレイヤーの準備完了を待たないこと。

- 侵入波のタイミングはシナリオ側で固定スケジュール（プレイヤー進捗非連動）
- 序盤は弱い侵入者が早い段階で来る → 最低限の防衛を優先するか構築を優先するかの判断
- 中盤以降は構築フェーズと防衛フェーズが重なる頻度を上げる
- 「全部屋完成してから防衛開始」というプレイを不可能にする
- **影響**: Phase 4 の侵入波生成と Phase 6 のシナリオ設計で、タイミング調整が最重要パラメータとなる

### 原則3: トレードオフの連続（Phase 5 に影響）

リソース（気）が常に不足し、全方面への投資が不可能であること。

- 気の収入（龍脈）は有限で、以下の用途が競合する:
  - 新しい部屋の掘削コスト
  - 既存部屋の強化
  - 仙獣の育成（気を消費して成長）
  - 仙獣の新規召喚コスト
  - 罠の設置・維持
- 1つに投資すると他が手薄になる構造を維持する
- 「全部の部屋がレベルMAX、全仙獣がレベルMAX」はゲームクリア前には到達不能なバランス
- **影響**: Phase 5 の経済バランスは「余裕がない」状態をデフォルトとして設計する

### アンチパターン（避けるべき設計）

- 十分な時間があれば完璧なダンジョンが完成する → ゲームの死
- 最適解が1つに収束し、毎回同じ配置が最強 → リプレイ性の喪失
- 侵入者が弱すぎて構築の邪魔にならない → 判断の必要性が消滅
- リソースが潤沢で全部に投資できる → トレードオフの消滅

### 検証方法

Phase 7（統合シミュレーション）で以下を検証する:

1. AIプレイヤーによる数千ゲームの自動実行で、「全部屋MAX」到達率がゲームクリア前に0%に近いこと
2. 同じシナリオでも地形RNGの違いで最適配置が変わること
3. 侵入波の50%以上が「構築中」（部屋掘削/仙獣配置の最中）に到達すること

### 備考

この判断はオリジナルのカオスシードのプレイ体験に基づく。具体的なパラメータ（地形障害物の密度、侵入波の間隔、リソースの逼迫度）はPhase 7のシミュレーションで調整する。重要なのはパラメータの値ではなく、3つの原則がシステム構造として組み込まれていることである。

# DECISIONS.md D002 追記分 — 人間から見たゲームの面白さの言語化

## 面白さの構造分解（プレイヤー体験の観点）

以下はオリジナルのカオスシードで実際に感じた面白さを、体験の種類ごとに言語化したもの。
システム設計の判断に迷ったときは、「この設計はどの面白さを生むのか、あるいは殺すのか」で検証する。

### 1. 計画する面白さ（構築フェーズ）

- 地形を見て「ここにこの部屋を作れば最高のレイアウトになるな」と構想する楽しさ
- 龍脈の位置と部屋の属性を考えて、相生の連鎖を頭の中で組み立てる知的快感
- 「理想の完成形」が見えている状態そのものが心地よい
- **設計への示唆**: 地形が読みやすく、計画を立てたくなる情報がプレイヤーに見えていること。風水の仕組みが直感的に理解できること

### 2. 妥協する面白さ（制約との対話）

- 理想通りにはいかない。地形の制約で「あ、でもこっちの部屋はこうなっちゃったんだ」と気づく瞬間
- でも悲観ではなく「じゃあ別の部屋をこう作ってこう繋げると8割ぐらいの満足度になるぞ」と代替案を見つける喜び
- 100点は取れないけど80点の解を見つける「パズルを解いている」感覚
- **設計への示唆**: 制約は「理不尽な壁」ではなく「別解を探す動機」であるべき。常に80点の妥協案が存在する地形生成が望ましい。0点（詰み）も100点（理想完遂）も起きてはいけない

### 3. 中断される面白さ（構築 × 防衛の衝突）

- チクチクとレイアウトを構築している最中に敵が攻めてくる
- 構築中だから防衛も中途半端。理想のレイアウトの途中なので理想の防御はできない
- この「両方とも不完全な状態で同時にやりくりする」忙しさが楽しい
- **設計への示唆**: 構築フェーズと防衛フェーズが明確に分離してはいけない。両方が重なるタイミングがゲームのピーク体験。シナリオ設計の最重要要素

### 4. 噛み合う面白さ（創発的成功）

- 不完全な防衛なのに、なんとか噛み合って撃退できた瞬間のカタルシス
- 仙獣の配置がたまたまいい位置にいた、部屋の属性がたまたま有利だった、そういう「計画半分・偶然半分」の成功
- 自分の判断が報われた感覚と、運が味方した感覚の混合
- **設計への示唆**: 戦闘結果に適度なランダム性が必要。100%計算で勝てる戦闘はつまらないし、100%運の戦闘は理不尽。「正しい配置をした上で、ダイスが味方する」バランス

### 5. 完璧が来ないことの面白さ（永続的不完全性）

- 完璧な部屋ができるとそこでゲームは終わる。判断が要らなくなるから
- 面白いのは「完璧を目指すプロセス」であって「完璧な状態」ではない
- ゲーム終了時に「もうちょっとやりたかった」「次はもっとうまくやれる」と思える状態が理想
- **設計への示唆**: ゲームクリア条件は「完璧なダンジョン完成」ではなく「N波の侵入を耐え凌ぐ」等の生存系。クリア時点でもダンジョンは未完成で、「次こそは」のリプレイ動機が残る

## 面白さの連鎖モデル

上記の5つは独立ではなく、以下の順序で連鎖する:

```
計画する → 制約に出会う → 妥協する → 構築を始める
    ↑                                       ↓
    └──── 「次こそは」のリプレイ動機 ←── 防衛で中断される
                                            ↓
                                    不完全な状態で噛み合わせる
                                            ↓
                                    撃退成功のカタルシス
                                            ↓
                                    でも完璧ではない → 次の計画へ
```

このループが1ゲーム内で何度も回り、かつゲーム間でも回ることがリプレイ性を生む。

## パラメータ調整時の判断基準

Phase 7 のシミュレーションやプレイテストで調整を行う際、以下を基準とする:

| 面白さの種類       | 壊れるサイン                         | 調整の方向                                |
| ------------------ | ------------------------------------ | ----------------------------------------- |
| 計画する面白さ     | プレイヤーが計画を立てる前に敵が来る | 最初の侵入波までの猶予を伸ばす            |
| 妥協する面白さ     | 常に理想配置が可能 / 常に詰む        | 地形制約の密度を調整                      |
| 中断される面白さ   | 構築と防衛が完全に交互               | 侵入波の間隔をランダム化・構築中に重ねる  |
| 噛み合う面白さ     | 戦闘が常に圧勝 / 常に惨敗            | 侵入者の強さ曲線とリソース供給を調整      |
| 完璧が来ない面白さ | ゲームクリア前に全MAXに到達          | リソースの総量を絞る / クリア条件を早める |

# PRD — chaosseed-core

## 1. プロダクト概要

chaosseed-core は、カオスシード（風水回廊記）にインスパイアされたダンジョン経営シミュレーションのコアメカニクスエンジンである。
純 Go で実装し、ゲームエンジンに依存しない決定論的なロジックライブラリとして設計する。

後続プロダクト:
- **chaosseed-sim**: CLI シミュレーター（コアエンジンを利用）
- **chaosseed-game**: Ebitengine ベースの GUI ゲーム（コアエンジンを利用）

## 2. 設計原則

- **決定論的**: 同一の RNG シード → 同一の結果。乱数は必ず `types.RNG` インターフェース経由
- **ゲームエンジン非依存**: 描画・入力・音声など外部依存なし。純粋なロジックのみ
- **標準ライブラリのみ**: 外部ライブラリへの依存を禁止
- **型安全**: `interface{}` / `any` の多用を避け、ドメイン型で表現する
- **テスタブル**: テストカバレッジ 80% 以上を目標

## 3. ドメインモデル

### 3.1 五行（Element）

ゲームの根幹をなす属性システム。木・火・土・金・水の5属性。

- **相生（Generates）**: 木→火→土→金→水→木（生み出す関係）
- **相克（Overcomes）**: 木→土→水→火→金→木（打ち克つ関係）

五行は部屋・仙獣・アイテム・風水すべてに影響する。

### 3.2 洞窟（Cave）

プレイヤーが経営するダンジョン。グリッドベースの2Dマップ。

- **Grid**: 2D セル配列。各セルは Rock / Corridor / RoomFloor / Entrance のいずれか
- **Room**: 矩形の部屋。タイプ・属性・気の容量を持つ
- **Corridor**: 部屋間を接続する通路。BFS で最短経路を生成
- **AdjacencyGraph**: 部屋をノード、通路をエッジとするグラフ

### 3.3 龍脈と風水（DragonVein / Feng Shui）

洞窟内の気の流れを管理するシステム。

- **龍脈（DragonVein）**: 入口から洞窟内へ気を運ぶ経路
- **気（Chi）**: 各部屋が蓄積するエネルギー。部屋の維持・仙獣の育成に必要
- **風水評価**: 部屋の配置・五行の相性・龍脈との接続に基づくスコアリング
- 相生関係にある部屋が隣接するとボーナス、相克関係だとペナルティ

### 3.4 仙獣（Senju）

部屋に配置するクリーチャー。

- 各仙獣は五行属性を持つ
- 部屋の属性と仙獣の属性の相性で成長速度が変わる
- 侵入者への防衛戦力
- 気を消費して成長・進化する

### 3.5 侵入システム（Invasion）

冒険者がダンジョンに侵入するイベント。

- 侵入者は入口から入り、部屋を探索しながら進行
- 仙獣による防衛戦闘
- 罠部屋による妨害
- 侵入の成否がダンジョンの評判・経済に影響

### 3.6 経済（Economy）

ダンジョン経営のリソース管理。

- 気の収支バランス（龍脈からの供給 vs 部屋・仙獣の消費）
- 部屋の建設・拡張コスト
- 侵入者が落とすアイテム・資源

## 4. フェーズ計画

### Phase 1: 洞窟マップシステム（world/）

グリッドベースの2Dマップ、部屋の定義と配置、通路生成、洞窟全体の管理、シリアライズ。

サブフェーズ:
- **1-A**: 共有型定義（types/）— Pos, Direction, Element, RNG, Tick
- **1-B**: グリッドとセル管理 — Cell, Grid
- **1-C**: 部屋の定義と配置 — RoomType, Room, 配置ロジック
- **1-D**: 通路の生成 — Corridor, BFS ベースの経路生成
- **1-E**: 洞窟全体の管理 — Cave, AdjacencyGraph
- **1-F**: シリアライズ — JSON 保存/復元
- **1-G**: 統合検証

### Phase 2: 風水システム（fengshui/）

龍脈の定義、気の流れシミュレーション、風水評価スコア。

- 龍脈（DragonVein）の経路計算
- 部屋ごとの気の蓄積・消費モデル
- 五行相性に基づく風水スコア評価
- 隣接部屋の属性ボーナス/ペナルティ

### Phase 3: 仙獣システム（senju/）

仙獣の定義、配置、成長、進化。

- 仙獣の種族・属性・ステータス定義
- 部屋への配置と属性相性
- ティックベースの成長シミュレーション
- 進化条件と進化処理

### Phase 4: 侵入システム（invasion/）

侵入者の生成、経路探索、戦闘解決。

- 侵入者の種類・強さ定義
- 入口からの経路探索 AI
- 仙獣 vs 侵入者の戦闘解決
- 罠・妨害の効果処理

### Phase 5: 経済システム（economy/）

リソースの収支管理、建設コスト。

- 気の供給量計算（龍脈ベース）
- 部屋・仙獣の維持コスト
- 建設・拡張の費用モデル
- 侵入結果による収入

### Phase 6: シナリオシステム（scenario/）

ゲームの進行管理、イベント、勝利条件。

- シナリオ定義（目標、制約、初期状態）
- イベントトリガーシステム
- 勝利/敗北条件の評価

### Phase 7: 統合シミュレーション（simulation/）

全システムを統合したゲームループ。

- ティックベースのメインループ
- 全サブシステムの更新順序管理
- 状態のスナップショットと巻き戻し
- CLI シミュレーター向けインターフェース

## 5. パッケージ依存関係

```
types ← world ← fengshui ← senju ← invasion ← economy ← scenario ← simulation
```

矢印の方向に依存する。`types` は他のパッケージに依存しない。循環依存は禁止。

## 6. 品質基準

- テストカバレッジ: 80% 以上
- `go vet ./...`: 警告なし
- `go test -race ./...`: データ競合なし
- 全エクスポート型・関数に GoDoc コメント
- ゲームデータは JSON で外出し、`embed` で埋め込み

## 7. ティックモデル

ゲーム内時間は `Tick`（uint64）で表現する。1ティックがシミュレーションの最小時間単位。

- 風水の気の流れ: 毎ティック更新
- 仙獣の成長: N ティックごと
- 侵入イベント: 確率的に発生（RNG 経由）
- 経済の収支: ティック単位で計算
